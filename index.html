The problem

Here's what actually
happens with a live model.

Suppose a provider updates a model between two runs. Your evals look fine. But the recorded response for the same canonicalized request input is different. Your observability tool shows that the call happened. It may not preserve tamper-evident evidence of the recorded output.

Run 1 — March 3

"Summarise the key risks of deploying LLMs in production in one sentence."

Key risks include non-deterministic outputs, prompt injection vulnerabilities, hallucinated facts, and the absence of tamper-evident logging.

response_hash: 2f1563cc8c0b7b71…
Run 2 — March 10 · same prompt

"Summarise the key risks of deploying LLMs in production in one sentence."

Primary risks are hallucination, data leakage, adversarial prompt injection, and unpredictable behaviour changes following silent model updates.

response_hash: c41d8e3b5f9a2201…

How it works

One import.
Minimal changes in supported integrations.

In supported integrations, adoption can be minimal. AELITIUM wraps the call path, hashes the canonicalized request input and recorded response, and writes a bundle you can verify offline.

1
Add one import

One line near the top of your existing code. In supported integrations, changes can be minimal. Provider API keys and normal runtime config still apply.

from aelitium import capture_openai
2
Bundle written to disk

SHA-256 of the request. SHA-256 of the response. A binding_hash links both. Optional Ed25519 signatures add signer authenticity when configured.

request_hash + response_hash
→ binding_hash = SHA256(request_hash || response_hash)
→ optional signature
3
Verify anywhere, offline

Hand the bundle to your auditor, your client, your lawyer. They can verify the bundle themselves offline, without calling your server.

aelitium verify-bundle ./evidence
→ STATUS=VALID
4
Catch silent model changes

Same request, different response? AELITIUM detects differences between recorded responses for the same canonicalized request input - by comparing bundle hashes, not log text.

aelitium compare ./run_a ./run_b
→ STATUS=CHANGED
Trust model

AELITIUM provides these guarantees:

- Any post-capture modification is detectable

AELITIUM does NOT guarantee:

- That the LLM actually produced the response

- That the response is correct or truthful

- That the provider did not change behavior

Verification is offline and does not trust:

- the runtime that produced the bundle

- the model provider

- your infrastructure

AELITIUM covers two cases:

- Tamper detection: modified recorded output

- Behavior drift: same canonicalized request input, different recorded response

Positioning

Observability tools
weren't built for this.

Langfuse, Helicone, LangSmith — great for debugging latency and costs. But they run on servers. They are not designed to provide offline, tamper-evident verification of recorded outputs months later.

Compliance

In regulated environments,
auditability and traceability requirements make this increasingly relevant.

If you're building AI into anything regulated, someone is going to ask you to preserve tamper-evident evidence of what was recorded. AELITIUM provides tamper-evident evidence that can be verified offline, making it suitable for audit and compliance workflows.

EU AI Act · Art. 12
Logging & traceability
Requires logged, traceable records of high-risk AI outputs. AELITIUM bundles are tamper-evident by design - modifications to packed evidence are detectable.
SOC 2 · CC7
System monitoring & integrity
SOC 2 auditors ask whether your logs could have been altered. Offline verification is the cleanest answer to that question.
ISO 42001
AI management auditability
Third-party auditors don't need access to your infrastructure. They get the evidence bundle and verify it offline.
NIST AI RMF · MG 2.2
Traceability of AI decisions
Full evidence bundle: canonicalized request input, recorded response material, binding hash, and optional Ed25519 signature when configured.
